# Git 智能提交工作流

这是一个高效的 Git 提交工具，在提交前自动分析代码改动，调用相关优化技能，并生成高质量的提交信息。

## 核心原则

**信任 AI 的智能判断**：您已经具备强大的代码分析能力，本工具旨在提供框架而非限制您的判断。

**高自由度设计**：React 检测、优化应用、提交信息生成都依赖上下文判断，您可以根据具体情况选择最佳方法。

**简洁高效**：只提供必要的工作框架，避免过度指导。

**并行优先**：Phase 2 中的任务彼此独立，应在同一轮响应中并行发起（多个 Agent 或工具调用）。只有存在数据依赖时才串行等待。某个任务被跳过时不需要等待——直接标记跳过，继续推进。

## 工作框架

创建一个简洁的任务列表来跟踪进度：

```
Git 提交工作流（三阶段模型）：

Phase 1 — 基础分析（串行，后续阶段的前提）
- [ ] 分析 Git 状态：检查可提交更改，分析改动内容和类型

Phase 2 — 独立任务（并行执行，每个任务仅依赖 Phase 1）
- [ ] 智能检测 React 代码 + 条件性优化
- [ ] Bugfix 验证：检测到 bugfix 时补充单测验证
- [ ] 代码理解测验
- [ ] CLAUDE.md 智能更新
- [ ] 版本自动升级

Phase 3 — 收尾流程（串行，等待 Phase 2 全部完成）
- [ ] 执行智能提交
- [ ] 任务同步（ai-todo）
- [ ] 提交总结
```

**调度原则**：
- Phase 2 的任务之间没有数据依赖，应尽可能在一轮响应中并行发起
- 需要用户交互的任务（如代码测验）与其他任务同时发起，等待用户回答期间其他任务已完成
- 任何 Phase 2 任务的失败或跳过不阻塞其他任务
- Phase 3 必须等待 Phase 2 全部完成（包括用户交互结束）

## 关键决策点

### 1. React 代码检测

**基本原则**：
- **自主判断**：基于代码特征而非硬编码规则进行分析

### 2. 优化技能调用

**基本原则**：
- **条件性调用**：根据代码分析结果决定是否调用优化技能
- **透明预览**：在应用优化前向用户展示建议的改动
- **用户确认**：获得用户同意后再应用修改

**可用技能**：
- "react best practice"：React 最佳实践优化（条件性调用）
- "code simple"：代码简化优化（建议总是调用）

**调用方式**：
- 使用 Skill 工具调用相关技能（例如：`Skill: "react best practice"`）
- 提供必要的文件路径和代码片段作为上下文
- 在应用优化前向用户展示建议的改动
- 获得用户确认后使用 Edit 工具应用优化
- 如技能调用失败，记录警告信息并跳过优化步骤（优雅降级）

### 3. Bugfix 验证

**目的**：bugfix 提交时，确保有测试覆盖了 bug 场景——"修了就要验"。

**触发条件**：commit type 为 `fix` 且项目有测试框架。否则跳过。

**核心流程**：找到或创建对应测试 → 补充 bug 场景用例 → 运行相关测试确认通过 → 测试文件加入暂存区。

**原则**：一个 bug 一个用例，不过度扩展。测试失败时报告详情，让用户决定是否继续提交。

### 4. 提交信息生成

**语言**：全部使用中文（type 标签除外）

**格式**：`type(scope): 业务描述 (技术说明)`

- **业务描述**：用用户/产品视角说"发生了什么"，简洁优先，一眼看懂
- **技术说明**：用括号补充技术实现，可省略

**type 选取**：
- `feat` 新功能 / `fix` 修复 / `perf` 性能 / `refactor` 重构
- `style` 样式 / `docs` 文档 / `chore` 杂项 / `test` 测试

**示例**：
```
feat(报告): 支持一键导出 PDF (新增导出 API + 前端按钮)
fix(登录): 修复登录后页面空白 (useEffect 缺少依赖导致重渲染)
perf(列表): 长列表滚动更流畅 (虚拟化渲染，DOM 节点从 500→30)
refactor(Auth): 简化鉴权逻辑 (合并重复的 token 校验分支)
chore: 升级依赖版本
```

**禁止**：
- 禁止英文描述（type 标签除外）
- 禁止堆砌技术词汇而忽略业务含义
- 禁止模糊措辞，如「优化代码」「更新逻辑」

### 5. 代码理解测验

**目的**：Vibe Coding 时代，开发者的核心价值是有效监督 AI 产出。测验聚焦"为什么"和"会怎样"，而非代码表面细节。

**基本原则**：
- **智能跳过**：简单改动（格式、重命名、配置、注释、依赖更新）直接跳过
- **监督者视角**：问"如果我要对这段代码负责，我需要理解什么？"
- **聚焦决策层**：设计权衡、系统影响、失败模式——AI 不会主动告诉你的东西
- **远离细节层**：语法、API 参数、变量名、代码位置——这些是 AI 擅长的，不需要人记

**出题指南**：
- 使用 `AskUserQuestion`，1-2 道场景判断题
- 用"如果...会怎样"或"为什么不..."的句式，考察判断力而非记忆力
- 示例：「选择 A 方案而非 B，牺牲了什么？」「如果上游返回异常，这段代码会怎样？」

**答错处理**：
- 解释正确答案，引用具体代码
- 用户可选择重新作答或确认已理解后继续
- 目的是培养监督直觉，不是考试

### 6. CLAUDE.md 智能更新

**目的**：让项目的 CLAUDE.md 始终准确反映当前状态，避免文档与代码脱节。

**触发判断**——以下任一情况需要评估更新：
- 新增或删除插件/功能模块
- 项目结构发生变化（目录新增、重命名、移除）
- 重要配置或依赖发生变更
- 核心工作流程或规范发生调整
- 版本号变更

**跳过条件**——以下情况直接跳过：
- 纯代码内部修改（bug fix、重构、性能优化），不影响对外接口或结构
- 仅修改样式、测试、注释
- CLAUDE.md 本身已是此次改动的一部分

**执行步骤**：
1. 读取现有 CLAUDE.md
2. 对比当前改动，找出与文档不一致或缺失的部分
3. 用 Edit 工具直接应用最小化更新

**更新原则**：
- **最小改动**：只更新受本次提交影响的部分，不"顺手"重构文档
- **保持风格**：沿用已有的语言、格式和结构
- **事实优先**：只写已经实现的内容，不写计划中的功能

### 7. 版本自动升级

**目的**：当一个功能/任务完整落地时，自动升级版本号，确保插件市场、依赖方能感知到最新变更。

**任务完成判断**——符合以下特征时视为任务完成：
- 提交信息 type 为 `feat`（新功能交付）
- 对话上下文中用户表达了"完成"、"搞定"、"上线"等意图
- 本次改动关闭了某个明确的需求或 issue
- 多次迭代后首次提交完整的功能闭环

**跳过条件**——以下情况直接跳过：
- type 为 `fix`、`chore`、`style`、`docs`、`test`、`refactor` 等非功能性提交
- 明显是中间过程提交（"WIP"、"临时"、"待续"等措辞）
- 本次改动已包含版本文件的修改

**版本文件查找优先级**：
1. 改动文件所在插件目录的 `.claude-plugin/plugin.json`（`version` 字段）
2. 项目根目录的 `package.json`（`version` 字段）
3. 其他发现的版本文件（如 `pyproject.toml`、`Cargo.toml`）

**升级规则**（语义化版本）：
- `feat`：升级 minor 版本（`1.2.x` → `1.3.0`）
- 带 breaking change 标记：升级 major 版本（`1.x.x` → `2.0.0`）

**执行步骤**：
1. 判断是否需要升级（见上方条件）
2. 定位版本文件，读取当前版本号
3. 计算新版本号
4. 用 Edit 工具更新版本文件
5. 将版本文件加入本次提交的暂存区（`git add`）
6. 在总结中说明版本从 X 升级到 Y

### 8. 任务同步

**目的**：提交完成后，将本次提交与任务管理系统同步，作为重要收尾步骤，保持开发进度可追踪。

**前置检查**：使用 Bash 工具执行 `which ai-todo` 检查是否可用，不可用则静默跳过。

**执行步骤**：
1. **了解任务全局**：先运行 `ai-todo tasks:tree` 查看完整任务树，理解当前项目结构和任务层级
2. **命令发现**：通过 `ai-todo --help` 查看当前可用命令，理解 ai-todo 从远端实时生成的 CLI 能力
3. **智能关联**：根据提交信息、分支名、改动内容，在任务树中找到最匹配的任务
4. **更新进度**：关联到具体任务后，更新进度、添加工作日志、或标记完成
5. **创建新任务**：如果没有关联任务，根据任务树找到合适的父任务/项目空间，**在正确的层级下创建**（不要在顶层随意创建）
6. **优雅降级**：任何 ai-todo 操作失败都不应阻断已完成的 Git 提交

### 9. 提交总结

工作流最后一步，输出一个表格让用户快速掌握本次提交全貌。表格应包含：各个任务的执行总结情况, 面向用户表达，不要讲非常细的技术点，同时如果是多 git 仓库，要分开说明下各个仓库的改动等。根据实际情况自行决定哪些列有值、哪些省略。
