# 本地拟真测试验证

智能分析代码改动，自动决定验证策略，在提交前快速发现问题。

## 核心原则

**信任 AI 的智能判断**：根据改动内容、项目类型、上下文自主决定验证策略。不依赖硬编码规则，您已经具备足够的分析能力来判断"该验什么、怎么验"。

**拟真优先**：验证要尽可能模拟真实运行环境。能跑起来的就跑起来，能启动服务的就启动服务。静态分析是底线，运行时验证是目标。

**快速反馈**：这不是 CI，是提交前的快速体检。跳过与本次改动无关的验证，聚焦变更影响范围，单步超时 60 秒，总计不超过 5 分钟。

## 工作框架

创建一个简洁的任务列表来跟踪进度：

```
本地拟真测试验证：
- [ ] **变更分析**：识别改动文件、改动类型、影响范围
- [ ] **项目环境探测**：识别项目类型、可用工具链、现有测试设施
- [ ] **验证策略制定**：选择验证手段组合，向用户展示验证计划
- [ ] **执行验证**：按依赖顺序执行，记录每步结果
- [ ] **结果报告**：汇总验证结果，给出通过/失败/警告判定
```

## 关键决策点

### 1. 变更分析

**基本原则**：
- 通过 `git diff` 和 `git status` 分析改动内容
- 将改动文件分类：前端组件、后端逻辑、配置文件、测试文件、文档、样式、依赖等
- 评估改动的"影响半径"——只改了样式 vs 改了核心业务逻辑，决定后续验证深度

**影响半径判断**：
- **低影响**：文档、注释、样式微调、变量重命名 → 轻量验证或跳过
- **中影响**：单个模块内的逻辑修改、新增工具函数 → 精准验证相关测试
- **高影响**：跨模块改动、依赖变更、配置修改、API 接口变更 → 全面验证

### 2. 项目环境探测

**基本原则**：
- 自主检测项目特征文件，判断项目类型和可用工具链
- 不预设项目类型，根据实际发现的文件做判断

**探测要点**：
- `package.json` 中的 scripts（test、lint、typecheck、dev、build 等）
- `tsconfig.json` / `jsconfig.json`（TypeScript 项目）
- `pytest.ini` / `pyproject.toml` / `setup.py`（Python 项目）
- `Cargo.toml`（Rust 项目）、`go.mod`（Go 项目）
- `.eslintrc` / `biome.json` / `.prettierrc`（格式/lint 工具）
- 已有测试文件的分布和框架（Jest、Vitest、pytest、cargo test 等）
- CI 配置（`.github/workflows`、`.gitlab-ci.yml`）——了解项目的标准验证流程

### 3. 验证策略制定

这是核心决策点。根据「变更类型 × 项目环境」从以下验证手段中选择最合适的组合：

| 验证手段 | 适用场景 | 说明 |
|---|---|---|
| 类型检查 | TS/Flow 项目有类型改动 | `tsc --noEmit` 或等效命令 |
| Lint 检查 | 有代码文件改动 | eslint/biome/ruff 等，只检查改动文件 |
| 单元测试 | 改动文件有对应测试 | 只跑相关测试，不跑全量 |
| 构建验证 | 改动可能影响构建 | 依赖、配置、入口文件变更时 |
| 开发服务器启动 | 前端组件/页面改动 | 启动 dev server，检查是否报错 |
| API 端点验证 | 后端路由/接口改动 | 启动服务 + curl 验证关键端点 |
| 导入检查 | 新增/删除/重命名文件 | 检查所有 import 引用是否有效 |
| 冒烟测试 | 没有现有测试但改了核心逻辑 | AI 写临时脚本验证关键路径 |

**策略制定原则**：
- 改动越小，验证越精准（改了一个组件就只跑那个组件的测试）
- 改动越大，验证越全面（跨模块改动要跑构建 + 类型检查）
- 有现成测试就用现成测试，没有时自行判断是否需要写临时验证
- **向用户展示验证计划，获得确认后再执行**

**展示格式**：
```
验证计划：
1. ✅ 类型检查 — tsc --noEmit（改动涉及 .ts 文件）
2. ✅ 单元测试 — vitest run src/utils/format.test.ts（改动文件有对应测试）
3. ⏭️ 构建验证 — 跳过（本次改动不涉及构建配置）
4. ⏭️ Dev Server — 跳过（非前端组件改动）

预计耗时 < 30s，是否执行？
```

### 4. 执行验证

**执行顺序**：静态检查先于运行时验证
1. 类型检查 / Lint（最快，问题发现率高）
2. 单元测试（中等耗时）
3. 构建验证 / 服务启动（最慢）

**执行原则**：
- 每步记录：命令、耗时、退出码、关键输出（截取前 50 行）
- 遇到失败**不立即中断**，标记失败后继续执行剩余步骤
- 致命错误除外（如构建完全崩溃导致后续步骤无意义）
- 启动的后台服务（dev server 等）验证完成后必须清理进程
- 单步超时 60 秒，超时视为警告而非失败

### 5. 结果报告

生成结构化验证报告：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  验证报告
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 类型检查：通过 (2.1s)
✅ ESLint：通过，0 warnings (1.3s)
❌ 单元测试：2 passed, 1 failed (3.5s)
   └─ FAIL: src/utils/format.test.ts > formatDate > 处理空值
⚠️ 构建验证：跳过（本次改动不影响构建）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
结论：发现 1 个测试失败，建议修复后再提交。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**结论分级**：
- **全部通过**：所有验证步骤通过，建议提交
- **有警告**：存在跳过或超时的步骤，可以提交但需注意
- **有失败**：存在验证失败，建议修复后再提交

**失败时的建议**：
- 引用具体的失败输出
- 如果 AI 能判断修复方向，主动给出修复建议
- 询问用户：是否需要帮助修复，还是确认后继续提交
